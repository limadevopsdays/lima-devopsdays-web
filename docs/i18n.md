# i18n — DevOpsDays Lima

Sistema TypeScript-first sin dependencias externas. Soporta **`es`** (default) y **`en`**.

---

## Cómo funciona

```
src/app/i18n/index.tsx           → Locale, LocaleProvider, hooks
src/app/data/mockContent.i18n.ts → Datos pesados (speakers, schedule, FAQ…)
src/app/components/**/i18n.ts    → Diccionario por componente
```

`LocaleProvider` envuelve toda la app. Al cambiar locale actualiza `<html lang="…">`.

El hook central:

```ts
export const useI18n = <T,>(dict: Record<Locale, T>): T =>
  dict[useContext(LocaleContext)]
```

TypeScript garantiza en compilación que **ningún string quede sin traducir** — si falta
una clave en `en`, el build falla.

---

## Hooks

| Hook | Retorna | Uso |
|---|---|---|
| `useLocale()` | `'es' \| 'en'` | Lógica condicional (SEO, etc.) |
| `useSetLocale()` | `(l: Locale) => void` | Toggle ES\|EN |
| `useI18n(dict)` | `T` | Strings del componente |

---

## Patrón por componente

```ts
// HeroSection/i18n.ts
import type { Locale } from '../../../i18n'

type HeroI18n = { title: string; cta: string }

export const heroI18n: Record<Locale, HeroI18n> = {
  es: { title: 'DevOpsDays Lima 2026', cta: 'Comprar Tickets' },
  en: { title: 'DevOpsDays Lima 2026', cta: 'Buy Tickets' },
}
```

```tsx
// HeroSection/index.tsx
const t = useI18n(heroI18n)
return <h1>{t.title}</h1>
```

---

## Contenido pesado (`mockContent.i18n.ts`)

Arrays de speakers, agenda, FAQ y venue viven aquí como `Record<Locale, T[]>`.

```ts
const keynoteSpeakers = useI18n(keynoteSpeakersI18n)  // KeynoteSpeaker[]
```

---

## Añadir una traducción

**Nuevo string en componente existente** → añadir campo al tipo + ES + EN en `i18n.ts`.

**Nuevo componente** → crear `i18n.ts` junto al componente con el patrón de arriba.

**Nota sobre `og:locale`:** se mantiene estático en `es_PE` — describe el locale
cultural del evento (Perú), no el idioma de la UI.

---

## ¿Por qué no usar una librería?

No cambiar por ahora. Las razones:

- **Solo 2 idiomas (ES/EN)** — sin variantes regionales complejas ni plurales ambiguos
- **Contenido estático** — fechas de evento, textos fijos, sin contadores dinámicos
- **TypeScript safety** — eliminar eso por JSON agregaría riesgo sin beneficio real
- **Cero overhead de bundle** — react-i18next pesa ~30 KB gzip; el enfoque actual pesa 0

**¿Cuándo sí cambiaría?** Si alguno de estos ocurre:

- Se suma un 3er idioma con plurales distintos (portugués, francés)
- Hay traducciones a cargo de personas externas que necesitan trabajar sin tocar código
- La cantidad de strings crece tanto que el mantenimiento manual se vuelve un problema

Si llegara ese momento, **Lingui** sería la mejor opción: es la única librería con
soporte TypeScript first-class similar al enfoque actual, y tiene un extractor CLI
que escanea el código automáticamente.
